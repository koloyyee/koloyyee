## Which Apps?
Last week I was searching about configuring the my laptop with Ansible, I did say I will use Terraform for deployment. However, after some research I could use Kubernetes(K8s) instead.

Before heading out to learn using K8s, I asked myself do I need k8s? and what kind of apps I'll need? The answer is I need the knowledge of using k8s with minimum overhead, and I want to hold an e-library and photo albums.

After some Googling and AI-ing, the answer is I will use [K3s](https://docs.k3s.io), then self-host [Immich](https://immich.app) and [Calibre-web](https://github.com/janeczku/calibre-web). I challenge myself to learn with proper domain routing and SSL and Kubernetes. This is where I found DevOps fun and practical!

## K3s
### What is K3s?
K3s is a lightweight Kubernetes distribution perfect for a single-node VPS. Real Kubernetes concepts, minimal overhead.

### Setting Up K3s
On the remote server:
```sh
curl -sfL https://get.k3s.io | sh -s - --write-kubeconfig-mode 644 --tls-san <tailscale_ip>
ufw disable 
sudo systemctl restart k3s
```
On your local machine:
```sh
mkdir -p ~/.kube
scp username@ip:/etc/rancher/k3s/k3s.yaml ~/.kube/config
```

### Edit config — replace server IP with your Tailscale IP
```sh
vi ~/.kube/config
# server: https://<tailscale_ip>:6443
brew install kubectl
kubectl get nodes
```
You now control a remote Kubernetes cluster from your Mac over Tailscale. The deployment architecture looks like this:
`Mac/Local machine (kubectl) → Tailscale → Homelab (K3s/containerd)`

## Deploying Calibre-Web
The first service — a good way to learn the basics before tackling something more complex.
```yml
# Which Kubernetes API group handles this resource type
apiVersion: apps/v1
# The type of resource — Deployment manages a set of identical pods
kind: Deployment
metadata:
  # Unique name for this Deployment within the namespace
  name: calibre-web
spec:
  # How many identical pod instances to run at all times
  replicas: 1
  selector:
    # The Deployment uses this label to find and manage its pods
    # Must match template.metadata.labels below
    matchLabels:
      app: calibre-web
  # The Pod blueprint — every pod the Deployment creates looks like this
  template:
    metadata:
      # Labels stamped onto every pod — must match selector.matchLabels above
      # Also how the Service finds this pod
      labels:
        app: calibre-web
    # The Pod spec — defines what runs inside the pod
    spec:
      containers:
      # Each item is one container inside the pod
      - name: calibre-web
        # The container image to pull and run
        image: lscr.io/linuxserver/calibre-web:latest
        # Environment variables injected into the container at runtime
        env:
        # Run the process as user ID 1000 (avoids running as root)
        - name: PUID
          value: "1000"
        # Run the process as group ID 1000
        - name: PGID
          value: "1000"
        # Timezone for the container's clock
        - name: TZ
          value: "America/Toronto"
        # LinuxServer mod that installs the full Calibre binary inside the container
        - name: DOCKER_MODS
          value: "linuxserver/mods:universal-calibre"
        ports:
        # The port the application listens on inside the container
        # Informational only — does not publish the port externally
        - containerPort: 8083
        # Mounts — connects named volumes to paths inside the container
        volumeMounts:
        # Mount the calibre-config volume at /config inside the container
        - name: calibre-config   # references volumes[].name below
          mountPath: /config
        # Mount the calibre-books volume at /books inside the container
        - name: calibre-books    # references volumes[].name below
          mountPath: /books
      # Volume sources — defines where the data actually comes from on the host
      # Every volumeMount above must have a matching entry here
      volumes:
      # Mounts a directory from the host filesystem into the pod
      - name: calibre-config
        hostPath:
          # Absolute path on the host (your Dell laptop/server)
          path: /home/loyyeeko/media/calibre-config
          # Directory must already exist on the host
          type: Directory
      - name: calibre-books
        hostPath:
          path: /home/loyyeeko/media/books
          type: Directory
---
# A separate resource — the Service that routes traffic to the pod
apiVersion: v1
kind: Service
metadata:
  name: calibre-web-service
spec:
  selector:
    # Finds pods with this label — matches template.metadata.labels above
    # This is how the Service knows which pods to send traffic to
    app: calibre-web
  ports:
  # The port the Service exposes internally within the cluster
  - port: 8083
    # The port on the pod/container to forward traffic to
    # Matches containerPort above
    targetPort: 8083
```
Apply and test:
```sh
kubectl apply -f calibre-web.yml
kubectl get pods
kubectl port-forward svc/calibre-web-service 8888:8083
```
Hit http://localhost:8888 to verify before setting up permanent domain routing.

## Deploying Immich
Immich is more complex — it needs Postgres, Redis, and the app server. The instinct coming from Docker Compose is to put everything in one Pod. Don't.
Each service gets its own resource:
* StatefulSet for Postgres
* Deployment for Redis
* Deployment for Immich server

Why StatefulSet for Postgres?
* Stable pod DNS name (immich-postgres-0)
* volumeClaimTemplates automatically provisions persistent storage
* Can restart independently without affecting other pods

## Configuration Files
#### Create secrets
Secrets first — never hardcode passwords:
```yml
apiVersion: v1
kind: Secret
metadata:
  name: immich-db-secret
type: Opaque
stringData:
  POSTGRES_PASSWORD: "your-password"
  POSTGRES_USER: "immich"
  POSTGRES_DB: "immich"
```

#### Configuring Postgres
Postgres StatefulSet:
```yml
apiVersion: apps/v1
kind: StatefulSet # Important for database, and don't forget serviceName below!
metadata:
  name: immich-postgres
spec:
  serviceName: immich-postgres # A must-have when using StatefulService
  replicas: 1
  selector:
    matchLabels:
      app: immich-postgres
  template:
    metadata:
      labels:
        app: immich-postgres
    spec:
      containers:
      - name: postgres
        image: tensorchord/pgvecto-rs:pg14-v0.2.0
        env:
        - name: POSTGRES_USER
          valueFrom:
            secretKeyRef:
              name: immich-db-secret
              key: POSTGRES_USER
        - name: POSTGRES_PASSWORD
          valueFrom:
            secretKeyRef:
              name: immich-db-secret
              key: POSTGRES_PASSWORD
        - name: POSTGRES_DB
          valueFrom:
            secretKeyRef:
              name: immich-db-secret
              key: POSTGRES_DB
        volumeMounts:
        - name: immich-db-data
          mountPath: /var/lib/postgresql/data
  volumeClaimTemplates:
  - metadata:
      name: immich-db-data
    spec:
      accessModes: ["ReadWriteOnce"]
      resources:
        requests:
          storage: 5Gi
---
apiVersion: v1
kind: Service
metadata:
  name: immich-postgres
spec:
  selector:
    app: immich-postgres
  ports:
  - port: 5432
    targetPort: 5432
```
Internal DNS — services talk to each other by Service name, not IP:
```yml
- name: DB_HOSTNAME
  value: "immich-postgres"
- name: REDIS_HOSTNAME
  value: "immich-redis"
```
[Find the rest of the files here]( https://github.com/koloyyee/homelab/tree/main/k3s )

Apply in order:

kubectl apply -f immich-postgres.yml

## Nginx Proxy Manager (NPM) for SSL + Domain Routing
NPM needs to bind ports 80 and 443 directly on the host. K3s ships with Traefik which holds those ports by default — disable it first.

#### Disable Traefik:

Create /etc/rancher/k3s/config.yaml on the server:
```yml 
disable:
  - traefik
```
```sh
sudo systemctl restart k3s
```
#### NPM with hostNetwork:
```yml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-proxy-manager
spec:
  replicas: 1
  selector:
    matchLabels:
      app: nginx-proxy-manager
  template:
    metadata:
      labels:
        app: nginx-proxy-manager
    spec:
      hostNetwork: true
      containers:
      - name: nginx-proxy-manager
        image: jc21/nginx-proxy-manager:latest
        ports:
        - containerPort: 80
        - containerPort: 443
        - containerPort: 81
        volumeMounts:
        - name: npm-data
          mountPath: /data
        - name: npm-letsencrypt
          mountPath: /etc/letsencrypt
      volumes:
      - name: npm-data
        hostPath:
          path: /home/loyyeeko/npm/data
          type: DirectoryOrCreate
      - name: npm-letsencrypt
        hostPath:
          path: /home/loyyeeko/npm/letsencrypt
          type: DirectoryOrCreate
```
`hostNetwork: true` bypasses Kubernetes networking entirely and binds directly to the host — no Service needed, no NodePort range restrictions.
Access the NPM admin UI at http://your-server-ip:81. Register your account.

#### Wiring Up Domains

Get ClusterIPs for your services:
```sh
kubectl get services
```
In NPM add proxy hosts:
* books.loyyee.co → 10.43.x.x:8083 (Calibre-Web ClusterIP)
* photos.loyyee.co → 10.43.x.x:2283 (Immich ClusterIP)
Enable Let's Encrypt SSL on each — NPM handles cert issuance and renewal automatically.

## Be Careful!
YAML indentation is unforgiving. volumes must be a sibling of containers under `spec.template.spec`. One level off and Kubernetes silently ignores the field — no error, just a pod that can't find its volume.
* `volumeMounts` and `volumes` are both required. Declaring a volume under `volumes` does nothing without a corresponding `volumeMounts` entry inside the container.
* `StatefulSet` requires `serviceName`. Without it `kubectl apply` throws a validation error.
* `apps/v1` not `app/v1`. Easy typo, hard to spot.
* `hostPath` data is safe across pod restarts. Deleting a pod or Deployment never touches files on the host filesystem.

## End State
```md
books.loyyee.co   → NPM → Calibre-Web (Deployment)
photos.loyyee.co  → NPM → Immich Server (Deployment)
                              ├── immich-postgres (StatefulSet)
                              └── immich-redis (Deployment)
```
All managed via kubectl from a Mac over Tailscale. No Docker Compose, no GUI — just YAML and a lightweight Kubernetes cluster on an idle laptop.
