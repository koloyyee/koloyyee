## Ansible
Before deploying my first app, I would like to share what and how I configure the server. I have been learning system administration, TDD, and QA automation from the [Obey the Goat Book](https://www.obeythetestinggoat.com/); and the best way to understand is by doing. Therefore, I start using [Ansible](https://docs.ansible.com/) for the server configuration. Unlike plain Bash scripts, Ansible is idempotent — it only makes changes when something is actually out of place.

## Inventory, Variables, and Playbook
Ansible is basically running a bunch of pre-scripted Bash scripts on your remote server. First make sure you installed [the Ansible CLI](https://docs.ansible.com/projects/ansible/latest/installation_guide/intro_installation.html) on your local machine. Once it is ready, we can create our first file, `inventory.yml`; we can define the hosts and special variables it can use later by Ansible.

Sample:
```yml
home_server: # Any name you like, it doesn't matter.
  hosts: # Ansible will look the particular host name under this scope.
    homelab: # This is THE host Ansible will look for later on in the playbook.yml
      ansible_host: "{{ tailscale_ip }}" # this is a variable.
      ansible_user: <username> # username when ssh to the remote server. e.g: ssh user1@123.45.67
      ansible_ssh_private_key_file: ~/.ssh/ssh_private_key # the private key allows your local machine to access the remote server.
```

### Who uses inventory.yml?
Once our `inventory.yml` is defined, we can create a `playbook.yml` and this is the "playbook" Ansible will act on. Here is an example to simply ping the remote server.

Sample:
```yml
- name: Homelab Setup Playbook # name of the playbook, doesn't matter
  hosts: homelab # The same name from inventory.yml. E.g: home_server: > hosts: > <unique_hostname>
  become: true # Run tasks as sudo/root
  tasks:
    # Sanity test
    - name: Ping my host # name of the task.
      ansible.builtin.ping: # what it will do.
```
### Real Works
Pinging the server does nothing useful. We will configure the server to upgrade `apt` packages and keep running even with the lid shut.
```yml
...
  tasks:
    [...]
    - name: Update apt cache and upgrade system packages
      apt:
        update_cache: yes
        upgrade: dist
        cache_valid_time: 3000

    - name: Configure system to keep running while lid closed
      lineinfile:
        path: /etc/systemd/logind.conf
        regexp: '^#?HandleLidSwitch=' 
        line: 'HandleLidSwitch=ignore' # keep laptop running
      notify: Restart systemd-logind
```
Then we will install some utility packages and restart the systemd-logind.
```yml
  tasks:
    [...]
    - name: Install utilities
      apt:
        name:
          - curl
          - git
          - ufw
          - acpi # check the battery percentage.
          - brightnessctl # set the laptop screen brightness
        state: present

    - name: Reduce laptop screen brightness to 1%
      ansible.builtin.command: brightnessctl set 1%

    - name: Restart system-logind
      systemd:
        name: systemd-logind
        state: restarted
```

## Create variable
If you remember there is `{{ tailscale_ip }}` in our `inventory.yml`, it is a variable but where does it come from? It is an encrypted variable. We can create it by running the following bash scripts, when `ansible-playbook` runs it will pick up the variable.
```sh
echo "tailscale_ip: <ip>" > ansible/group_vars/all/inventory_vault.yml
ansible-vault encrypt ansible/group_vars/all/inventory_vault.yml
```
For plain variables you can follow [this guide](https://docs.ansible.com/projects/ansible/latest/playbook_guide/playbooks_variables.html).

## Create encrypted password
In our `playbook.yml` we set `become: true`, which tells Ansible to run tasks as `sudo`. To supply that sudo password securely, we use a vault.

1. Create password with vault
```sh
ansible-vault create group_vars/all/vault.yml # create password with ansible-vault cli
```
2. Store the vault password in a separate file so Ansible can unlock it automatically
```sh
echo "secret_password" >> ~/.vault_pass
chmod 600 ~/.vault_pass # granting read 
```

## Run it!
Here are a few ways to run `ansible-playbook` depending on where the password was stored.

1. Manually type the password
```sh
ansible-playbook -i inventory.yml playbook.yml --ask-become-pass
```

2. Access from vault
```sh
ansible-vault view group_vars/all/vault.yml # create an encrypted password
ansible-playbook -i inventory.yml playbook.yml --vault-password-file --ask-vault-pass # only enter the vault password, ansible will access the encrypted password.
```
3. Access from separate location
```sh
ansible-playbook -i inventory.yml playbook.yml --vault-password-file ~/.vault_pass
```

## One more thing!
We can move `inventory.yml` and `playbook.yml` into the `ansible` directory:
```sh
mkdir ansible
mv inventory.yml playbook.yml ansible/
```
If you `cd ansible` then run `ansible-playbook -i inventory.yml playbook.yml --ask-vault-pass` it should still work!
